## Deferred to b10
These cohere as a single theme: re-institute a CLI context, push DI/bootstrap orchestration closer to the domain/service-provider layer, add a logging value object, and split async feature execution.
### 1. Re-institute a CliContext (lean the CLI blueprint)
The CLI blueprint currently carries helper logic that belongs in a context. Reintroduce a CLI context (extending `AppInterfaceContext`) that retrieves CLI commands, converts them into contexts, and builds/parses the argument parser, removing the blueprint helper functions.
- **Files:** `tiferet/blueprints/cli.py` (e.g. `build_argument_kwargs`), `tiferet/blueprints/tiferet_cli.py`.
- **Also fixes:** the `from ..mappers import CliCommandAggregate` import inside `tiferet/blueprints/tiferet_cli.py` — mappers should not be reachable from a blueprint; this cleanup ties directly to the new CLI context.
- **Review refs:** https://github.com/greatstrength/tiferet/pull/841#discussion_r3409697156 , https://github.com/greatstrength/tiferet/pull/841#discussion_r3409794191
- **Note:** consider whether CLI command → context conversion belongs to a domain method or the new context (per reviewer's open question).
### 2. Service provider owns DI assembly
Move the DI type-map/constant assembly out of `DIContext` and into the service provider, modeled like a domain event: the `DIService` (ServiceRepository) becomes a direct dependency of the provider, and the assembly operations become methods on it. This pulls the process closer to the domain layer instead of the interface (contexts/assets/blueprints).
- **Files:** `tiferet/contexts/di.py` (`DIContext`), `tiferet/di/` (service provider).
- **Review ref:** https://github.com/greatstrength/tiferet/pull/841#discussion_r3416510166
### 3. Bootstrap `default_*` redesign + asset restructuring
Remove the execute-time `default_*` parameters from events and replace them with a cleaner custom-bootstrapping approach: rearrange the asset modules into an id-keyed, dict-like structure (similar to how YAML data is shaped) and feed each record into its domain object via `**kwargs` with the retrieved id as the appropriate id. Decide the final placement of default-handling (event vs. context vs. blueprint) and document the new bootstrap structure.
- **Affected event params:** `GetFeature.execute(default_feature_index=...)` (`tiferet/events/feature.py`), `ListAllSettings.execute(default_config_index=..., default_constants=...)` (`tiferet/events/di.py`), `ListCliCommands.execute(default_commands_list=...)` (`tiferet/events/cli.py`), `GetAppInterface.get_from_defaults(...)` (`tiferet/events/app.py`).
- **Assets:** `tiferet/assets/{app,feature,cli_commands,logging}.py` (now plain dict literals after b9) — restructure into id-keyed mappings.
- **Review refs:** https://github.com/greatstrength/tiferet/pull/841#discussion_r3416826979 , https://github.com/greatstrength/tiferet/pull/841#discussion_r3416834724 , https://github.com/greatstrength/tiferet/pull/841#discussion_r3416859194 , https://github.com/greatstrength/tiferet/pull/841#discussion_r3416860176
- **Note:** item 3 overlaps item 1 (the `default_commands_list` question ties to the CLI context) and item 2 (service-provider-owned config assembly). Sequence after items 1–2.
### 4. LoggingConfiguration value object + GetLoggingConfiguration event
Introduce a `LoggingConfiguration` value object returned by a new `GetLoggingConfiguration` event that resolves a `Logger` plus its associated `Formatter`/`Handler`. `LoggingContext` consumes the value object instead of assembling raw logging configs inline.
- **Files:** `tiferet/contexts/logging.py`, `tiferet/events/` (new event), `tiferet/domain/logging.py` (value object).
- **Review ref:** https://github.com/greatstrength/tiferet/pull/841#discussion_r3416795673
### 5. Split AsyncFeatureContext + add `Feature.is_async`
Split the async methods out of `FeatureContext` into an `AsyncFeatureContext` subclass via inheritance, preserving the existing method names (with the async suffix). Add an `is_async` flag to the `Feature` domain object and use it in `AppInterfaceContext` during the on-demand context-loading step to select the sync vs. async feature context.
- **Files:** `tiferet/contexts/feature.py` (split `handle_feature_step_async` / `execute_feature_async` into `AsyncFeatureContext`), `tiferet/domain/feature.py` (`is_async` field), `tiferet/contexts/app.py` (select context by `Feature.is_async`).
- **Review ref:** https://github.com/greatstrength/tiferet/pull/841#discussion_r3416701898
- **Note:** the b9 work already renamed the async method to `handle_feature_step_async`, so the names are ready for the split.
